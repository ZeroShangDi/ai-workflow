# 质量规范 — 充满信心地交付

以下原则涵盖如何验证正确性、沟通变更以及保持代码库长期健康。当工作从"做完了"推进到"上线了"时应用。

## 测试

### 测试金字塔

按每单位投入的最大信心分配测试投入：

- **单元测试（~70%）**：快速、隔离、每次变更时运行。测试纯逻辑——函数、composables、store actions。能立即捕获回归。
- **集成测试（~20%）**：验证模块协同工作。测试 store 交互、针对 mock 端点的 API 调用、多组件工作流。
- **端到端测试（~10%）**：验证关键用户流程。保持少量并聚焦——它们缓慢、脆弱且维护昂贵。

### 测试行为，而非实现

测试应验证代码*达成了什么*，而非它*内部怎么做*。如果重构不需要修改任何现有测试就能通过，这些测试就是设计良好的。如果重命名一个私有函数导致 10 个测试失败，这些测试耦合在实现细节上——它们测错了东西。

### 测试命名：场景 → 预期结果

命名测试，让失败能立即传达哪里出了问题：`"用户未激活时，登录返回 UNAUTHORIZED"`。读者无需打开测试体就能理解设定、动作和预期。

### 逻辑明确时使用 TDD

对于解析器、校验器、计算和状态转换：
1. 写一个失败的测试，表达期望的行为
2. 写最小代码使其通过
3. 以测试为安全网进行重构

TDD 产生的代码天然可测试——测试是接口的第一个消费者。探索性工作、UI 润色或需求不明确时跳过。

### 测试覆盖率哲学

覆盖率百分比是信号，不是目标。聚焦覆盖*行为*：正常路径、边缘情况（null、空、边界值）、错误路径（网络失败、无效输入）和状态转换。100% 覆盖率但没有边缘情况测试，比 70% 覆盖率但测了所有危险路径更糟糕。

## 代码审查

### 审查工具无法发现的东西

工具处理格式化（Prettier）、语法（TypeScript）和 lint 规则（ESLint）。人工审查聚焦：
- **正确性**：边缘情况处理了吗？错误路径能工作吗？
- **意图**：读代码能否清楚理解其目的？
- **架构**：这和周边代码协调，还是和现有模式冲突？
- **安全性**：XSS、注入、数据泄露——浏览器扩展中尤其关键

### 每一行都要审查

没有"太简单不需要审查"的变更。看似微不足道的变更可能在扩展上下文中（content script ↔ background ↔ sidepanel）产生令人意外的交互。

### 以正确心态审查

目标不是找东西批评——而是确保变更是正确的、可理解且可维护的。如果没发现问题，这就是一次成功的审查。

## 提交

### 约定式提交

```
<type>: <简短的中文描述>

[可选的正文，解释为什么，而非做了什么]
```

类型：`feat`（新功能）、`fix`（修复）、`refactor`（重构，不改变行为）、`style`（格式化）、`docs`、`chore`、`perf`

### 小且单一关注的提交

每个提交只做一件事。小提交容易审查（审查者能在脑中装下整个变更）、容易回滚（无附带伤害）、数月后在 git blame 中也容易理解。

### 提交信息解释动机

diff 展示了*什么*。信息解释*为什么*——问题、约束、权衡。读 git log 的人不用打开任何文件就能理解项目的演进。

### 如果 pre-commit hook 失败

修复问题，然后创建一个新的提交。不要 amend——上一个提交根本没完成，所以没有可 amend 的。Amending 已发布的提交会重写共享历史；永远不要这么做。

## 技术债务

### 使其可见

临时方案需要标签：`// TODO(用户名): <需要改什么> <日期>`。无归属、无日期的 `// TODO` 对所有人都不可见，包括六个月后的作者本人。

### 排期修复

没有进入 backlog 的债务永远不会被处理。当你走了捷径，创建一个任务。被追踪的债务是可控的；悄无声息在代码中累积的债务是危险的。

### 语义化版本意识

当修改共享代码（包、API、导出类型）时：
- 新增向后兼容的功能 → 次版本号
- 破坏性变更 → 主版本号
- Bug 修复 → 修订号

在破坏之前，想想消费者。

## 文档

### 代码首先自我文档化

清晰的命名、小函数和显式的数据流是首要文档。只有当各部件之间的*交互*无法通过单独阅读每个部件来理解时，才需要独立文档。

### 为复杂交互写文档

当理解某个模块需要跨多个文件、上下文或异步边界追踪数据时——多步骤流程、浏览器扩展中的跨上下文消息传递、含大量转换的状态机——`docs/<module>.md` 是必要的。

### 文档随代码存放

文档和代码在同一个仓库，和变更在同一个提交中更新。过时的文档有主动危害——它误导读者以为自己理解了一个已经变了样的系统。

## 可观测性

### 有目的地打日志

- `error`：需要立即人工关注
- `warn`：出乎意料但已处理——值得后续调查
- `info`：关键生命周期事件（会话开始、功能开关、认证状态变更）
- `debug`：详细状态，仅用于本地开发

每行日志都应回答诊断问题时有人真正会问的问题。为打日志而打日志只会制造噪音，掩埋信号。

### 性能有基线

关键路径（消息渲染、SSE 处理、API 调用）应有已知的性能画像。当你在关键路径上改代码时，考虑：这会增加重渲染吗？阻塞主线程吗？随时间累积内存吗？

### 生产环境错误监控

生产错误应自动上报——不要等用户投诉。关键错误（认证失败、数据丢失、支付问题）应触发告警。不知道有问题，就没办法修复。
