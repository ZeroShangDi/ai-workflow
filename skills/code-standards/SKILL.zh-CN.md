# 编码规范 — 写出好代码

以下原则适用于函数、变量和文件层面——即编写代码时逐行逐句应该做的决策。它们旨在让代码易于理解、安全修改。

## 函数设计

### 一个函数只做一件事

如果你需要用"和"或"或者"来描述一个函数做了什么，就拆分它。一个函数应该只做一件事，且这件事能用一句简短的话说清楚。

### 保持函数简短

目标控制在 20 行以内。长函数难以测试、难以理解——读者脑子里需要同时装太多东西。当函数变长时，提取私有辅助函数。调用方保持可读；提取出来的部分可以独立测试。

### 限制参数数量

纯函数最多接受 3 个参数。超过这个数量，将参数归组到一个 options 对象中。这避免了调用处参数顺序混乱的问题，也让后续扩展不构成破坏性变更。

### 优先使用纯函数

纯函数——相同输入永远得到相同输出，无副作用——便于测试、易于推理。将副作用（API 调用、文件 I/O、DOM 写入）推到边缘层：事件处理器、服务层或 store actions。核心逻辑保持纯粹。

### 保持一致的返回类型

一个函数不应该在一个分支返回对象，在另一个分支返回 null，在第三个分支抛出异常。选择一种表示"无结果"的模式（null、undefined、Result 类型或异常），并在整个模块中一致使用。

### 按意图命名，而非按实现

调用方关心*做什么*，而非*怎么做*。`saveUser()` 即使你换了数据库也照样正确；`insertIntoDatabase()` 换了数据库就名不副实了。改了实现，名字应该依然合理。

## 命名

### 名称即文档

一个好名字可以省去注释。如果你需要写注释来解释某个变量存了什么，把它重命名。

- 函数：动词 + 业务名词 — `getUserInfo`、`validateEmail`、`computeDiscount`
- 布尔值：`is` / `has` / `can` 前缀 — `isActive`、`hasPermission`、`canEdit`
- 事件处理器：`handle` + 动作 + 目标 — `handleSendClick`、`handleFileDrop`

### 用有表现力的变量替代复杂表达式

将条件提取到命名变量，让意图一目了然：

```typescript
// 读起来像解谜
if (user.status === 'active' && user.role === 'admin' && !user.suspended)

// 读起来像自然语言
const isActiveAdmin = user.status === 'active' && user.role === 'admin' && !user.suspended
if (isActiveAdmin)
```

### 魔术数字用常量

除 0 和 1 之外的任何字面量都值得一个名字。`const TAX_RATE = 0.13` 自解释；`price * 0.13` 让读者猜。

### 用枚举替代魔术字符串

`Status.Active` 享有 IDE 自动补全、重命名安全性，编译期就能捕获拼写错误。`'active'` 可能因为拼成了 `'actvie'` 导致运行时 bug。

## 错误处理

### 明确处理，不要沉默

错误必须被捕获并处理。静默吞掉错误，会把可检测的失败变成下游的神秘 bug。错误信息应携带足够的上下文，让人无需调试器就能定位问题。

### 按类型分类错误

- **系统性错误**（网络断开、认证过期、5xx）→ 全局处理器（toast 提示、重试、日志）
- **业务错误**（无效输入、重复记录、4xx）→ 调用处附近的局部处理器

这样避免了到处散落重试逻辑，同时让业务规则违规的处理靠近相关 UI。

### 信任边界

校验跨越系统边界的数据（用户输入、API 响应、第三方数据）。信任内部代码之间的数据。不要在每个内部函数中重复校验同一个东西——这制造噪音并产生虚假的安全感。

## 防御性编程

### 避免 null 地狱

不要写 `if (data && data.user && data.user.name)`。使用可选链（`data?.user?.name`）和空值合并（`data?.user?.name ?? 'Unknown'`）。更好的做法是设计不需要 null 检查的数据结构——在边界处使用空对象或合理默认值。

### 在边界处校验，只校验一次

数据进入系统时进行解析和校验。此后，内部代码可以假设数据是有效的。在每个函数中分散校验只会增加维护负担和产生不一致的错误信息。

## 代码风格

### 正向条件判断

写 `if (isValid)` 而非 `if (!isInvalid)`。双重否定迫使读者在脑中翻转两次。正向形式处理更快，误解更少。

### 注释解释为什么，而非做了什么

代码展示了它做了什么。注释应解释不明显的原因：某个库 bug 的变通方案、一个令人意外的性能权衡、一条法规约束。如果"做了什么"不明显，改进命名，不要加注释。

### 空行创造视觉结构

将相关行归组；用空行分隔逻辑块。读者扫视文件时应能把握其结构，无需逐行阅读。

### 按来源组织 import

用空行分隔各组：第三方包 → 内部模块（`~/`）→ 相对导入 → 样式。让读者一眼可见外部依赖。

### 保持文件大小均衡

目标 100–300 行。低于 100 行显得零碎；超过 500 行意味着该拆分了。文件太大会隐藏结构；文件太小增加导航开销。

## 性能意识

### 对昂贵的纯函数做缓存

当纯函数以相同参数被频繁调用，且计算代价不低时，缓存结果。在 Vue 中是 `computed`；在纯 TS 中，使用有大小限制的 `Map`。

### 惰性求值

不要在实际需要之前计算值。Vue 的 `computed` 默认是惰性的。对于昂贵的初始化，使用惰性 getter 模式，而不是在模块加载时计算。

### 节流与防抖

- **防抖（debounce）**：等用户停止后再执行（边输入边搜索）
- **节流（throttle）**：每个时间间隔最多执行一次（滚动处理器、resize 监听器）

选错会导致 UI 卡顿（搜索用节流）或做无用功（滚动用防抖）。

### 批量更新状态

同一同步代码块中的多次状态变更应触发一次重渲染，而非多次。Vue 默认在同一个 tick 内批量处理。避免不必要地用 `await` 穿插状态变更——每个 `await` 创建一个新的微任务边界。

### 深拷贝 vs 浅拷贝

对不可变更新使用浅拷贝（展开运算符）——速度快，结构共享节省内存。只有真正需要完全隔离时才用深拷贝。对于复杂的嵌套更新，使用 Immer 等库，而不是手写深拷贝。

## Vue 3 专项

### 在 `<script setup>` 中按关注点分组

将同一功能的所有代码——状态、计算属性、方法、生命周期——放在一个代码块中。不要把相关逻辑散落在文件中。当一个关注点增长到约 50 行以上时，将其提取为 `useXxx` 组合式函数。

### 副作用放在 `watch`/`watchEffect` 中，而非 `computed` 中

计算属性应该是纯推导。绝不要在 computed 内部触发 API 调用、状态变更或 DOM 写入——这会破坏可预测性并导致无限更新循环。

### 对整体替换的大对象使用 `shallowRef`

当你整体替换对象而非修改嵌套字段时，`shallowRef` 可以避免深层响应式追踪的成本。同样适用于整体替换更新的长列表。

### 给列表使用稳定的 key

在 `v-for` 中始终使用稳定、唯一的标识符作为 `:key`——业务 id 或 uuid，绝不要用数组索引。当列表重排序时，基于索引的 key 会导致 DOM 回收 bug。
