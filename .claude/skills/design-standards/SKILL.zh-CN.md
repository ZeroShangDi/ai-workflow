# 设计规范 — 架构好系统

以下原则适用于跳出单行代码、思考整体结构时——模块、组件、store 和数据如何协同工作。它们让系统在增长过程中保持复杂度可控。

## 架构

### 组合优于继承

继承创建了僵化的层级——改了父类，所有子类都必须适应。组合让你从独立、可替换的部分组装行为。

在 Vue 3 中：优先使用 `useXxx()` 组合式函数而非基础组件类。优先使用插槽而非组件继承。仅在真正的"是一个"关系中保留继承，且层级控制在 2 层以内。

### 三次法则

- 第一次出现：直接写
- 第二次出现：复制，心里记下重复
- 第三次出现：*此时*才提取共享抽象

过早抽象比重复更糟糕。抽象得过早会把错误的形状锁定——你尚不知道什么真正会变化。到第三个用例时，模式就清晰了。

### 单向数据流

数据单向流动：`用户操作 → 状态更新 → 视图重渲染`。子组件通过事件通知父组件；绝不要直接修改 props。这让数据流可预测——当出现问题时，沿单一路径回溯即可。

### 副作用与纯逻辑分离

组件应负责渲染 UI 并委托给服务层。业务逻辑、API 调用、缓存和持久化属于专门的层——组合式函数、store actions 或服务模块。组件聚焦于展示；逻辑变得可独立测试和替换。

### 接口隔离

模块不应依赖任何它不用的东西。组件 props 应只包含组件实际需要的数据——不要因为组件只需渲染名字就传入整个 user 对象。窄接口意味着需求变更时的爆炸半径更小。

### 迪米特法则（最少知识原则）

对象只需与直接邻居通信。链式访问如 `user.profile.address.city` 将调用方耦合到整个对象图——改变任何一环，调用方就挂了。将遍历封装在方法后面：`user.getCity()` 隐藏了内部结构并吸收变更。

### 显式优于隐式

依赖应该可见（作为参数传递，而非从全局拉取）。逻辑应该可读（不要玩需要三遍才能理解的聪明技巧）。数据流应该可追踪（不要在深层调用栈中修改共享状态）。当出现问题时，你应该能顺着线索查下去。

### 行为局部性

相关的代码放在一起。私有辅助函数放在其调用方下面。一个功能的文件放在同一个目录。好的组织方式的衡量标准：修一个 bug 或加一个功能触及的文件尽可能少。

### 契约式设计

每个函数文档化其约定条款：
- **前置条件**：调用前必须满足的条件（在调用处校验）
- **后置条件**：返回后保证的条件（函数对调用方的承诺）
- **不变量**：执行过程中始终保持的条件

TypeScript 类型是契约执行的第一道防线——用它们表达约束，而不仅仅是形状。

### 配置与代码分离

可能无需改代码就需变更的值——URL、阈值、功能开关、超时时间——属于配置，不应硬编码。如果你发现自己想"这个后面可能要调"，现在就抽出来。

## 数据建模

### 单一数据源

每份数据有且只有一个权威所在。其他所有位置引用或派生自它——绝不保留自己的副本。当数据同时存在于两个地方时，其中一个迟早会出错。

### 派生状态不存储

如果能从现有状态计算出来，就不要存储。`fullName` 派生自 `firstName + lastName`；单独存储意味着每次 name 更新都必须同步更新 `fullName`——迟早有一条路径会被遗漏。在 Vue 中用 `computed`，在 Pinia 中用 getters。

### 不可变优先

共享状态应被替换，而非就地修改。更新时创建新版本。好处：每个变更都可追踪，撤消/重做很简单（恢复之前的引用），并发操作不会互相破坏。

### 状态机替代布尔标志

不要用 `isLoading`、`hasError`、`isEmpty` 等独立布尔值，建模明确状态：`idle | loading | success | error | empty`。这消除了不可能的组合（加载中 且 出错？），让每个状态转换明确且有意义。

### 规范化嵌套数据

深层嵌套让更新复杂、查找缓慢。使用扁平的关系结构：

```typescript
// 嵌套 — 难以更新单篇文章
{ users: [{ id: 1, posts: [{ id: 10, title: "..." }] }] }

// 规范化 — O(1) 查找，单点更新
{
  users: { byId: { 1: { id: 1, postIds: [10] } }, allIds: [1] },
  posts: { byId: { 10: { id: 10, title: "..." } }, allIds: [10] }
}
```

### 避免过大的状态对象

包含 10+ 个不相关字段的单一状态对象难以理解，还会导致不必要的重渲染。按领域拆分——聊天 store 不应包含 UI 偏好设置，反之亦然。

### 领域模型 ≠ DTO

- **领域模型**：承载业务规则和行为，存在于应用程序内部
- **DTO（数据传输对象）**：用于 API 通信的纯数据载体

将它们分开。在 API 边界处转换。不要让传输格式污染你的业务逻辑。

### 乐观更新带回滚

对于高频交互，立即更新 UI（乐观地），然后向服务端确认。如果请求失败，回滚到之前的状态。回滚必须完整且经过测试——部分回滚比等待服务端更糟糕。

### 稳定标识符

记录的唯标识符应一次分配且永不改变。使用业务 ID 或 UUID——绝不使用数组索引或临时 key。改了 ID 会破坏所有引用、缓存和指向该记录的关系。

### 给持久化数据加版本号

跨会话存储的数据（localStorage、IndexedDB）都应携带版本号。读取时运行迁移，将旧格式转换为当前格式。否则，一次数据格式变更就会破坏每个现有用户的存储状态。

## 设计模式

### 策略模式替代 if-else 链

当行为按类型变化时，使用查找映射替代级联判断：

```typescript
// 避免
if (type === 'A')
  handleA()
else if (type === 'B')
  handleB()

// 推荐 — 新增类型无需改动已有代码
const handlers: Record<string, () => void> = { A: handleA, B: handleB }
handlers[type]?.()
```

### 适配器模式用于外部依赖

将第三方 API 包装在自己的接口后面。当库升级或你换了提供商时，只需改适配器——其余代码不动。这对支付、存储和分析集成尤其有价值。

### 观察者模式用于解耦事件

发布者发事件而不知道谁在监听；订阅者响应而不知道谁在发。在 Vue 中，这意味着 `watch`/`watchEffect` 和事件总线。当多个独立组件需要响应同一状态变更时使用。

### 工厂模式用于复杂对象创建

当构建一个对象需要多步骤、多依赖或多决策时，将复杂性封装在工厂函数中。调用方拿到即用对象，无需知晓构建细节。

### 装饰器模式用于横切关注点

在不修改原函数的情况下添加行为（日志、计时、缓存、权限检查）。高阶函数和 Vue 的 `onServerPrefetch`/`watch` 在精神上类似装饰器。

## Vue 3 + Pinia 专项

### 状态只提升到必要的层级

多个组件共享的状态放到最近的公共祖先（provide/inject 或 Pinia store）。只被一个组件使用的状态留在该组件内。越低越好；越高越需要理由。

### 按领域拆分 store

一个 store 应该拥有一个领域——聊天消息、用户偏好、文件上传是独立的 store。不要创建单体"app store"。这让 actions 保持专注，减少重渲染范围。

### Actions 封装变更

组件调用 actions；actions 更新状态。绝不要从组件直接修改 store 状态。这个单一入口让校验、副作用和日志能在不触及组件代码的情况下实现。

### 对大块数据退出深层响应式

当 store 持有大块、不常变更的数据结构时，使用 `shallowRef` 或 `markRaw` 避免深层响应式的成本。通过整体替换对象来变更，而不是深入修改嵌套字段。
