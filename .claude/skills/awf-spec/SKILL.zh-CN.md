# AWF 规范 — 自治工作流引擎

此 skill 定义自治工作流系统的完整规范。`awf-run` 命令引用此 skill，不内联这些细节。

## 状态机

```
                              ┌──────────────────────────┐
                              │          PLAN            │
                              │  Q&A → 需求文档 → 原型    │
                              │  → WBS → 任务清单        │
                              └────────────┬─────────────┘
                                           │
                         ┌─────────────────┼─────────────────┐
                         │ 无 UI            │ 有 UI            │ 简单任务
                         ▼                 ▼                  │
                   ┌──────────┐    ┌──────────────┐           │
                   │   CODE   │    │    DESIGN    │           │
                   │ loop 逐个 │    │ 3 风格 → 选一 │           │
                   │ 执行任务  │    │ → 逐步生成 UI │───────────┘
                   └────┬─────┘    └──────┬───────┘
                        │                 │ UI 就绪
                        │◀────────────────┘
                        │
         ┌──────────────┼──────────────┐
         │ 遇到 bug      │ 任务完成      │ 阶段性完成
         ▼               ▼              ▼
   ┌──────────┐   ┌──────────┐   ┌──────────┐
   │  DEBUG   │   │  REVIEW  │   │  FINISH  │ ← 里程碑标识
   └────┬─────┘   └────┬─────┘   └────┬─────┘
        │               │              │
        │ 定位根因       │ 审查通过      │ 下一里程碑
        ▼               ▼              ▼
     回 CODE      ┌──────────┐   回 PLAN / CODE
                  │   TEST   │
                  └────┬─────┘
                       │
            ┌──────────┼──────────┐
            │ 代码缺陷   │ 测试通过   │ 文档过期
            ▼           ▼           ▼
       回 CODE/DEBUG ┌──────────┐ ┌──────────┐
                     │  COMMIT  │ │   DOC   │ ← 任意节点可触发
                     └────┬─────┘ └────┬─────┘
                          │              │
                          ▼              │ 更新完成
                     ┌──────────┐        │
                     │  FINISH  │◀───────┘
                     └──────────┘
```

## 关键规则

- **DOC 任意节点触发** — 不固定在特定位置；可在编码前/后或测试期间触发
- **FINISH = 里程碑标识** — 不是项目终点；复杂任务经过多个 FINISH 循环
- **DEBUG 是中断驱动** — 当任何阶段（CODE/REVIEW/TEST）遇到问题时触发
- **TEST → DOC 自动** — 测试期间检测到文档过期自动触发 `/w-doc`，然后重新测试
- **允许多次 COMMIT** — 每个需求可在每个里程碑内提交一次或多次

## 倒计时暂停机制

在阶段转换暂停点，系统在自动推进前显示倒计时。这允许用户在需要时介入，同时保持工作流在正常操作中持续运行。

### 概念

```
阶段 A 完成
    │
    ▼
┌─────────────────────────────────┐
│ ⏳ 将在 N 秒后自动进入阶段 B     │
│ 回复任意内容即暂停工作流          │
└─────────────────────────────────┘
    │
    ├── 倒计时内用户回复 → 工作流暂停，保存断点
    │
    └── 倒计时结束无回复 → 自动进入阶段 B
```

### 暂停点及默认时长

| 暂停点 | 默认（秒） |
|--------|----------|
| PLAN → CODE/DESIGN | 15 |
| DESIGN → CODE | 10 |
| COMMIT 前 | 10 |
| FINISH → 下一里程碑 | 15 |

### 行为

- **标准模式**：在每个暂停点显示倒计时。用户可回复打断并保存断点。之后通过 `/awf-run --resume` 恢复。
- **`--auto` 模式**：倒计时为 0 — 无暂停，立即自动推进。
- **打断处理**：用户在倒计时期间回复时，工作流暂停，当前状态保存至 `.claude/awf-state.json`，并告知用户如何恢复。

### 配置

`.claude/user/pause-config.json` 中的用户配置：

```json
{
  "pauseCountdown": {
    "enabled": true,
    "duration": 15
  }
}
```

`duration` 是所有暂停点的默认值。后续可扩展每个暂停点的独立时长配置。

### 实现说明

CLI 中无法实现实时动态倒计时 UI。使用 `ScheduleWakeup` 模拟：在 `duration` 秒后调度唤醒。如果用户在此期间已回复，取消自动推进。如果没有，则进入下一阶段。

## 文档体系

| 层级 | 内容 | 位置 | 阶段 |
|------|------|------|------|
| 项目级 | 需求文档、原型、WBS、任务清单 | `docs/<feature>.md` | PLAN |
| 需求级 | 需求文档 `.md` | `docs/<module>/<req-id>.md` | CODE |
| 需求级 | 测试用例 `.test.md`（721 金字塔） | `docs/<module>/<req-id>.test.md` | CODE |
| 需求级 | 开发日志 `.log.md` | `docs/<module>/<req-id>.log.md` | CODE |
| 临时 | 审查结果 | `temp/`（可随时删除） | REVIEW/TEST |

## 状态持久化

每次状态转换、任务完成、Issue 创建时将状态写入 `.claude/awf-state.json`。

```json
{
  "task": "原始任务描述",
  "currentState": "CODE",
  "currentMilestone": 1,
  "plan": {
    "reqDoc": "docs/<feature>.md",
    "wbs": [{ "id": "1", "deliverable": "...", "acceptance": "...", "deps": [] }],
    "tasks": [{ "id": "1", "desc": "...", "wbsRef": "1.1", "deps": [], "parallel": false, "status": "done" }]
  },
  "milestones": [{ "id": 1, "desc": "...", "status": "active" }],
  "lastUpdated": "2026-06-29T10:00:00Z"
}
```

**恢复**：
1. `/awf-run --resume` 或新的 `/awf-run` 检测到已有状态文件
2. 显示：上次中断位置 + 进度摘要 + 待决策 Issue
3. 用户选择：继续 / 重新开始

## 阶段详情

### PLAN — 需求对齐

1. 分析任务，探索代码库
2. 交互式 Q&A（AskUserQuestion），每次 1-3 个问题
3. 依次产出：需求文档 → 原型 → WBS → 任务清单
4. 推进前与用户确认
5. 涉及 UI → 进入 DESIGN；纯逻辑 → 进入 CODE

### DESIGN — UI 风格选择（可选）

PLAN 标记"需要 DESIGN 阶段"时触发：
1. 读取 `.claude/user/style-preferences.md` 获取历史偏好
2. 生成 3 个视觉风格各异的方案
3. 用户通过 AskUserQuestion 选择
4. 保存至 `.claude/user/style-preferences.md`
5. 逐步生成 UI — 每次一页/一个组件，逐一确认
6. UI 就绪 → 进入 CODE

### CODE — 同步循环执行

1. 选取下一个 `status: pending` 且依赖已满足的任务
2. 执行：w-dev → w-review → w-test → w-commit 循环
3. 任务完成 → 标记 `done` → 下一个任务
4. 里程碑所有任务完成 → 里程碑 FINISH
5. 所有里程碑完成 → 项目 FINISH

**异常处理**：
- 设计缺陷 → 回 PLAN，更新状态
- 简单 bug → 进入 DEBUG → 修复 → 回 CODE
- 需要人工介入 → 创建 Issue 到 `.claude/issues/`，继续其他任务
- 任务拆分不当 → 动态合并/拆分/新增任务，更新状态
- 技术阻塞 → 标记 blocked，记录原因，继续其他任务

**测试策略**：默认仅产出测试用例文档（无测试代码）。仅在以下情况添加测试代码：(a) 逻辑过于复杂，仅靠文档无法验证；或 (b) 频繁回归导致 token 成本不可持续。

### REVIEW

1. 审查当前任务的所有变更
2. 聚焦 quality-standards 维度：正确性、安全性、性能、可维护性
3. 结果写入 `temp/review-<req-id>.md`
4. 严重问题 → 回 CODE；无问题 → 进入 TEST

### TEST

1. 运行 `pnpm test` 检查存量测试
2. 运行 `pnpm typecheck`
3. 对照 `docs/<module>/<req-id>.test.md` 验证
4. 结果写入 `temp/test-<req-id>.md`
5. 全部通过 → COMMIT；代码缺陷 → 回 CODE/DEBUG；文档过期 → 自动 w-doc → 重新测试

### COMMIT

1. 分析变更，生成 Conventional Commits 消息
2. 每个任务允许多次提交
3. 禁止 Co-Authored-By 签名
4. 更新 `<req-id>.log.md` 记录提交信息
5. 提交前倒计时暂停（遵循倒计时暂停机制；`--auto` 模式下跳过）

### FINISH — 里程碑收尾

1. 质量检查：lint + typecheck + debug 残留
2. 性能分析：仅限当前里程碑变更
3. 文档同步：检查并更新相关文档
4. Issue 扫描：汇总 `.claude/issues/` 中的待决策 Issue
5. 输出里程碑摘要
6. 更新状态：标记里程碑完成
7. 下一里程碑 → 回 PLAN；全部完成 → 结束；发现问题 → 回 CODE

## Issue 升级

当需要人工介入时：

**AI 创建 Issue** 到 `.claude/issues/ISSUE-<N>-<简短描述>.md`，包含：
1. 问题描述（现象、触发条件、影响）
2. 当前上下文（阶段、任务、相关文件、已尝试步骤）
3. 方案 A（推荐）— 描述 + 影响 + 风险
4. 方案 B — 描述 + 影响 + 风险
5. 用户方案槽 — 留空供用户填写

**AI 轮询**每次状态转换时：扫描 `.claude/issues/`，检查已决策但未执行的 Issue，汇总待决策 Issue。

## 自主决策权限

**无需询问的操作**（直接执行）：
- 代码风格 / 命名 → 遵循 ESLint/Prettier 和 code-standards
- 文件组织 → 遵循现有项目结构
- 依赖选择 → 优先复用
- 类型标注 → 严格 TS，禁止 `any`
- 错误处理 → 显式，无静默错误
- 文档过期 → 自动 w-doc
- 审查通过 → 自动推进到测试
- 测试通过 → 自动推进到提交
- Issue 检查 → 每次状态转换
- 状态文件写入 → 每次状态转换

**必须暂停的操作**（倒计时自动推进；`--auto` 模式下完全跳过）：
- PLAN Q&A 之后 / PLAN 产出之后
- DESIGN 风格选择 / 每个 UI 页面之后
- 新增第三方依赖
- 破坏性 API 变更
- 每个任务 COMMIT 前
- 存在两种同等有效方案的模糊权衡
- 里程碑 FINISH 之后

以上每个暂停点均遵循倒计时暂停机制：显示倒计时，允许用户打断或自动推进。

## 目录结构

```
project/
├── .claude/
│   ├── awf-state.json          # 工作流状态（持久化）
│   ├── user/                   # 用户个性化（不提交）
│   └── issues/                 # Issue 升级文档
├── docs/
│   ├── <feature>.md            # 项目级（PLAN 产出）
│   └── <module>/
│       ├── <req-id>.md         # 需求文档
│       ├── <req-id>.test.md    # 测试用例（721）
│       └── <req-id>.log.md     # 开发日志
└── temp/                       # 临时文件（可删除）
```
