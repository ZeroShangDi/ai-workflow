# 代码审查

对指定的代码变更进行审查，发现问题并提出改进建议。

## 参数

`$ARGUMENTS` — 审查目标（如 `当前分支改动`、`src/sidepanel/api.ts`、`PR #123`）。留空则审查当前未提交的所有变更。

## 前置 Skill 调用

自动调用 `superpowers:requesting-code-review` skill — 按标准化流程执行审查。

## 关联 Skill

审查时以 **code-standards**（微观质量：函数、命名、错误处理）和 **quality-standards**（宏观质量：测试、架构、安全）为双重标准。

## 执行流程

1. **获取变更**：
   - 无参数：`git diff` 查看当前所有未暂存改动 + `git diff --cached` 查看已暂存改动
   - 指定文件：`git diff -- $ARGUMENTS`
   - 指定 PR：通过 `gh pr diff` 获取
   - 指定分支：`git diff main...$ARGUMENTS`

2. **逐文件审查**，关注以下维度：
   - **正确性**：逻辑是否正确，边界情况是否处理
   - **安全性**：是否引入 XSS、敏感信息泄露等风险
   - **性能**：是否有不必要的重渲染、内存泄漏、大循环
   - **可维护性**：命名是否清晰、职责是否单一、是否有重复代码
   - **风格一致性**：是否符合项目的 ESLint / Prettier 规范
   - **浏览器扩展特有**：权限是否最小化、消息通信是否正确、manifest 是否需要更新

3. **输出审查报告**：

```
## 代码审查报告

### 概要
- 涉及文件：X 个
- 问题数量：严重 X / 建议 X

### 问题列表

#### [严重/建议] 文件名:行号 — 问题标题
问题描述
建议修改方式

...

### 总结
[整体评价和建议]
```

## 审查规则

- 不强制要求补充测试（除非改动涉及核心逻辑且无任何测试覆盖）
- 不建议无实际收益的重构
- 关注实际 bug 和安全风险，减少代码风格层面的挑剔（已有 lint 覆盖）
