# 测试用例排查

根据功能模块的 `.test.md` 测试用例文档，逐个排查每条测试用例在代码中的实际表现。

## 参数

`$ARGUMENTS` — 功能模块名称（如 `agent`、`content-scripts`、`remote-config`），对应 `docs/$ARGUMENTS.test.md`

## 关联 Skill

测试用例设计和排查标准遵循 **quality-standards** skill 中的测试金字塔、行为测试原则和命名规范。

## 执行流程

1. **加载测试文档**：读取 `docs/$ARGUMENTS.test.md`，解析所有 TC 用例
2. **逐条排查**：按编号顺序，对每个 TC 执行：
   - 阅读对应的源码文件，定位测试用例描述的代码路径
   - 验证**前置条件**中引用的变量、state、配置是否仍存在且名称正确
   - 验证**操作**步骤描述的函数调用/事件触发路径是否与当前代码一致
   - 验证**期望状态**描述的返回值、副作用、状态变化是否与代码逻辑匹配
   - 判定结果：`✅ 通过` / `⚠️ 文档过期` / `❌ 代码缺陷`

3. **输出排查报告**：

```
## 测试用例排查报告 — $ARGUMENTS

### 统计
- 总用例数：X
- ✅ 通过：X
- ⚠️ 文档过期：X（文档描述与代码不一致，需更新文档）
- ❌ 代码缺陷：X（代码未满足用例期望，疑似 bug）

### 详细结果

#### ✅ TC-1.1.1 [场景名]
代码路径：`src/path/file.ts:行号`
验证通过

#### ⚠️ TC-1.2.3 [场景名]
代码路径：`src/path/file.ts:行号`
差异：文档描述 [A]，实际代码行为 [B]
建议：更新文档 / 确认是否为有意变更

#### ❌ TC-2.1.1 [场景名]
代码路径：`src/path/file.ts:行号`
问题：期望 [A]，但代码逻辑会导致 [B]
根因分析：[为什么会出现这个问题]
修复建议：[具体的修改方式]

### 建议操作
- 文档过期项：执行 `/w-doc $ARGUMENTS` 更新文档
- 代码缺陷项：执行 `/w-debug [问题描述]` 进一步排查
```

## 排查规则

- **代码为准**：排查的目的是验证代码行为是否符合文档预期，不是机械跑测试
- **追踪完整链路**：跨上下文通信（background ↔ content script ↔ sidepanel）需要沿整个链路验证
- **关注边界用例**：边界值、异常路径、幂等性用例是最容易出问题的
- **区分文档过期和代码缺陷**：如果代码改动是合理的重构/优化，则标记为文档过期；如果代码确实不满足业务需求，则标记为代码缺陷

## 注意

- 如果 `docs/$ARGUMENTS.test.md` 不存在，提示用户先执行 `/w-doc $ARGUMENTS` 生成
- 排查过程中发现的代码缺陷不要直接修复，只报告（避免排查过程引入新问题）
- 如果用例数量较多（>20），可以分批排查，每批输出中间报告
