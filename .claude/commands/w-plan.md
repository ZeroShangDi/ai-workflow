# 计划制定

为指定任务制定详细实施方案，通过交互式 Q&A 对齐需求，产出需求文档、原型、WBS 和任务清单。

## 参数

`$ARGUMENTS` — 任务描述（如 `添加暗黑模式`、`重构 Agent 通信层`）

## 关联 Skill

方案设计时以 **design-standards** skill 为架构准则。

## 执行流程

### 步骤 1：需求理解

分析 `$ARGUMENTS`，快速探索相关源码理解现状。

### 步骤 2：交互式 Q&A 对齐

**不要一次性问完所有问题**。每次提出 1-3 个最关键的问题，使用 `AskUserQuestion` 逐步对齐。目标是这个阶段确定 **60-70% 的细节**，最低要求是大方向正确。

提问优先级：
1. **目标边界**：做什么、不做什么
2. **用户场景**：谁用、怎么用、核心路径
3. **技术约束**：兼容性要求、性能要求、依赖限制
4. **验收标准**：怎么判断做完了、做到什么程度

直到用户表示「可以开始产出」或关键方向已对齐，进入下一步。

### 步骤 3：产出

按**顺序**产出以下四份文档：

#### 3.1 需求文档
输出到 `docs/<feature-name>.md`：

```markdown
# [功能名称] 需求文档

## 背景与目标
[为什么要做，解决什么问题]

## 用户场景
[核心使用路径，1-3 个典型场景]

## 功能范围
- 包含：[明确边界]
- 不包含：[明确排除]

## 验收标准
- [ ] 标准 1
- [ ] 标准 2
```

#### 3.2 产品原型
用 ASCII 或 Mermaid 描述关键交互流程和页面结构：

```
页面结构：
┌─────────────┐
│  Header      │
├─────────────┤
│  Main       │
│  ┌─────────┐│
│  │ Content  ││
│  └─────────┘│
└─────────────┘

交互流程：
用户操作 → 状态变化 → UI 响应
```

如涉及前端 UI，标记「需进入 DESIGN 阶段」。

#### 3.3 WBS（工作分解结构）
**面向交付物**的层级分解，在任务清单之前生成：

```
1. [交付物 A] — 可独立验证的产出
   1.1 [子交付物 A1]
   1.2 [子交付物 A2]
2. [交付物 B]
   2.1 [子交付物 B1]
   2.2 [子交付物 B2]
```

每个节点标注：
- **交付物**：具体产出是什么（文档/模块/组件/接口）
- **验收方式**：怎么验证完成
- **依赖**：依赖哪些其他交付物

#### 3.4 任务清单
从 WBS 推导出的**可执行任务列表**，按依赖排序：

```markdown
## 任务清单

| # | 任务 | 对应 WBS | 依赖 | 预估 |
|---|------|---------|------|------|
| 1 | [任务描述] | 1.1 | - | - |
| 2 | [任务描述] | 1.2 | 1 | - |
| 3 | [任务描述] | 2.1 | 1, 2 | - |
```

**并行标记**：互不依赖的任务标记为 `[P]`，可在开发阶段并行执行。

### 步骤 4：确认

输出完整方案后暂停，等待用户确认。确认后存入 state 文件进入下一阶段。

## 与 awf-run 的协作

被 awf-run 调用时，产出结构化数据写入 `.claude/awf-state.json`：

```json
{
  "currentState": "PLAN",
  "plan": {
    "summary": "...",
    "reqDoc": "docs/<feature-name>.md",
    "wbs": [ ... ],
    "tasks": [ ... ]
  }
}
```
