# 自治工作流执行

接收复杂任务描述，按状态机模型自主推进完整开发流水线。支持交互式需求对齐、风格选择、loop 循环执行、断点恢复、Issue 升级和多里程碑交付。

## 参数

| 形式 | 说明 |
|------|------|
| `/awf-run <任务描述>` | 标准模式，在关键决策点暂停确认 |
| `/awf-run --auto <任务描述>` | 快捷模式，跳过所有确认点 |
| `/awf-run --resume` | 从上次中断处恢复 |

## 关联 Skill（全程生效）

- **design-standards** — 规划/设计阶段：架构设计、数据建模、状态管理
- **code-standards** — 编码阶段：函数设计、命名、错误处理、代码风格
- **quality-standards** — 验证/交付阶段：测试、审查、提交、文档

## 状态机模型

```
                              ┌──────────────────────────┐
                              │          PLAN            │
                              │  Q&A → 需求文档 → 原型    │
                              │  → WBS → 任务清单        │
                              └────────────┬─────────────┘
                                           │
                         ┌─────────────────┼─────────────────┐
                         │ 无 UI            │ 有 UI            │ 简单任务
                         ▼                 ▼                  │
                   ┌──────────┐    ┌──────────────┐           │
                   │   CODE   │    │    DESIGN    │           │
                   │ loop 逐个 │    │ 3 风格 → 选一 │           │
                   │ 执行任务  │    │ → 逐步生成 UI │───────────┘
                   └────┬─────┘    └──────┬───────┘
                        │                 │ UI 就绪
                        │◀────────────────┘
                        │
         ┌──────────────┼──────────────┐
         │ 遇到 bug      │ 任务完成      │ 阶段性完成
         ▼               ▼              ▼
   ┌──────────┐   ┌──────────┐   ┌──────────┐
   │  DEBUG   │   │  REVIEW  │   │  FINISH  │ ← 里程碑标识
   └────┬─────┘   └────┬─────┘   └────┬─────┘
        │               │              │
        │ 定位根因       │ 审查通过      │ 下一里程碑
        ▼               ▼              ▼
     回 CODE      ┌──────────┐   回 PLAN / CODE
                  │   TEST   │
                  └────┬─────┘
                       │
            ┌──────────┼──────────┐
            │ 代码缺陷   │ 测试通过   │ 文档过期
            ▼           ▼           ▼
       回 CODE/DEBUG ┌──────────┐ ┌──────────┐
                     │  COMMIT  │ │   DOC   │ ← 任意节点可触发
                     └────┬─────┘ └────┬─────┘
                          │              │
                          ▼              │ 更新完成
                     ┌──────────┐        │
                     │  FINISH  │◀───────┘
                     └──────────┘
                          │
                     ┌────┴────┐
                     │ 下一里程碑 │
                     └─────────┘
```

### 关键规则

- **DOC 可在任意节点触发** — 不固定位置，编码前/编码后/测试时均可
- **FINISH 是里程碑标识** — 不是项目终点，复杂任务经过多次 FINISH
- **DEBUG 是中断驱动** — CODE/REVIEW/TEST 任一阶段出现问题即切入
- **TEST → DOC 自动** — 测试发现文档过期时自动更新后重测
- **COMMIT 可多次** — 每个需求 commit 一次或多次，一个里程碑内可有多个 commit

## 文档体系总览

| 层级 | 内容 | 位置 | 产出阶段 |
|------|------|------|---------|
| 项目级 | 需求文档、原型、WBS、任务清单 | `docs/<feature>.md` | PLAN |
| 需求级 | 需求文档 `.md` | `docs/<module>/<req-id>.md` | CODE |
| 需求级 | 测试用例 `.test.md`（721） | `docs/<module>/<req-id>.test.md` | CODE |
| 需求级 | 开发日志 `.log.md` | `docs/<module>/<req-id>.log.md` | CODE |
| 临时 | 审查结果 | `temp/`（可随时删除） | REVIEW/TEST |

## 状态持久化

整个执行过程中，状态自动写入 `.claude/awf-state.json`：

```json
{
  "task": "原始任务描述",
  "currentState": "CODE",
  "currentMilestone": 1,
  "plan": {
    "reqDoc": "docs/<feature>.md",
    "wbs": [
      { "id": "1", "deliverable": "...", "acceptance": "...", "deps": [] }
    ],
    "tasks": [
      { "id": "1", "desc": "...", "wbsRef": "1.1", "deps": [], "parallel": false, "status": "done" },
      { "id": "2", "desc": "...", "wbsRef": "1.2", "deps": [1], "parallel": false, "status": "active" }
    ]
  },
  "milestones": [
    { "id": 1, "desc": "...", "status": "done" },
    { "id": 2, "desc": "...", "status": "active" }
  ],
  "lastUpdated": "2026-06-23T11:00:00Z"
}
```

**恢复流程**：
1. `/awf-run --resume` 或新开 `/awf-run` 检测到已有 state 文件
2. 展示上次中断位置 + 进度摘要 + 待决策 Issue 列表
3. 用户选择：继续执行 / 重新开始

**自动保存时机**：每次状态转换、任务完成、Issue 创建时写入。

## Phase 1: PLAN（需求对齐与规划）

**目标**：大方向确定 60-70%，产出四份结构化文档。

### 1.1 需求理解

分析 `$ARGUMENTS`，快速探索相关源码。

### 1.2 交互式 Q&A

使用 `AskUserQuestion`，每次提出 1-3 个最关键问题。逐步对齐：

1. **目标边界**：做什么、不做什么
2. **用户场景**：核心使用路径
3. **技术约束**：兼容性、性能、依赖
4. **验收标准**：如何判断完成

**直到关键方向已对齐**（用户确认或表达满意），进入产出步骤。

### 1.3 顺序产出

#### a) 需求文档 → `docs/<feature-name>.md`
背景与目标、用户场景、功能边界、验收标准。

#### b) 产品原型
ASCII 或 Mermaid 描述关键交互流程和页面结构。涉及 UI 时标记「进入 DESIGN 阶段」。

#### c) WBS（工作分解结构）
**面向交付物**的层级分解：

```
1. [交付物 A] — 可独立验证
   1.1 [子交付物 A1] — 验收方式 / 依赖
   1.2 [子交付物 A2] — 验收方式 / 依赖
2. [交付物 B]
   ...
```

#### d) 任务清单
从 WBS 推导的**可执行任务列表**，按依赖排序。互不依赖的任务标记 `[P]`（可并行）。

| # | 任务 | 对应 WBS | 依赖 | 并行 |
|---|------|---------|------|------|
| 1 | ... | 1.1 | - | - |
| 2 | ... | 1.2 | 1 | - |
| 3 | ... | 2.1 | - | [P] |

### 1.4 确认

输出完整方案后**暂停确认**。确认后判断：有 UI → 进入 DESIGN；纯逻辑 → 进入 CODE。

---

## Phase 2: DESIGN（UI 风格选择与设计，可选）

**触发条件**：PLAN 标记「需进入 DESIGN 阶段」。

### 2.1 读取偏好

读取 `.claude/user/style-preferences.md`，有历史偏好则以此为默认。

### 2.2 生成 3 种风格

生成 3 种有显著差异的 UI 风格方案，每种包含：配色、布局、组件风格、字体、适合场景。

### 2.3 用户选择

`AskUserQuestion` 展示风格方案，用户选择一项。

### 2.4 保存偏好

将选中风格写入 `.claude/user/style-preferences.md`。

### 2.5 逐步生成 UI

根据 PLAN 产出的原型，**逐个页面/组件**生成 UI 设计。每完成一个暂停确认，不一次性生成全部。

### 2.6 转入 CODE

UI 设计就绪后，转入 CODE 阶段。

---

## Phase 3: CODE（编码执行 — loop 同步模式）

**当前版本仅支持同步模式**。从任务清单中逐个提取就绪任务，按 CODE → REVIEW → TEST → COMMIT → FINISH 闭环执行。

### 3.1 任务调度

1. 从 state 文件中取下一个 `status: pending` 且依赖已满足的任务
2. 启动该任务的开发闭环
3. 任务闭环完成 → 标记 `done` → 取下一个
4. 当前里程碑所有任务 done → 进入里程碑 FINISH
5. 全部里程碑 done → 项目 FINISH

### 3.2 单个任务闭环

```
选取任务 → w-dev 编码 → w-review 审查 → w-test 验证 → w-commit 提交
                ↑              │               │
                └── DEBUG ─────┘               │
                ↑                              │
                └──────────── 缺陷 ─────────────┘
```

每个任务完成后，产出三份文档（调用 `/w-doc <module>/<req-id>`）：
- `<req-id>.md` — 需求文档
- `<req-id>.test.md` — 测试用例（721 分层）
- `<req-id>.log.md` — 开发日志（核心变更、关键决策）

### 3.3 测试策略

- **默认**：只留测试用例文档，不写测试代码
- **补充测试代码的触发条件**：
  - 用例无法被正确判定（逻辑过于复杂，文档判断不了对错）
  - 频繁触发回归导致 token 消耗过大（用代码一次性验证）
- 测试代码放在 `src/__tests__/` 对应位置

### 3.4 遇到问题时的处理

| 情况 | 处理方式 |
|------|---------|
| 方案缺陷 | 回到 PLAN 调整方案，更新 state |
| 简单 bug | 切入 DEBUG → 定位 → 回 CODE 修复 |
| 需人工介入 | 创建 Issue 到 `.claude/issues/`，继续执行其他任务 |
| 任务拆分不合理 | 动态调整（合并/拆分/新增任务），更新 state |
| 技术阻塞（无法继续） | 记录阻塞原因，标记任务为 blocked，继续其他 |

---

## Phase 4: REVIEW（审查）

当前任务编码完成后进入审查。

1. 审查当前任务的全部变更
2. 关注 **quality-standards** skill 维度：正确性、安全性、性能、可维护性
3. 审查结果写入 `temp/review-<req-id>.md`（可随时删除）
4. 严重问题 → 回 CODE 修复后重新 REVIEW
5. 无严重问题 → 进入 TEST

---

## Phase 5: TEST（测试验证）

1. 运行 `pnpm test` 确认存量测试通过
2. 运行 `pnpm typecheck` 确认类型完整
3. 读取对应 `docs/<module>/<req-id>.test.md`，逐条验证
4. 测试结果写入 `temp/test-<req-id>.md`（可随时删除）

**结果**：
- 全部通过 → 进入 COMMIT
- 代码缺陷 → 回 CODE/DEBUG
- 文档过期 → 自动 `/w-doc <module>/<req-id>` → 回到 TEST

---

## Phase 6: COMMIT（提交）

1. 分析当前任务变更
2. 生成 Conventional Commits 格式 message
3. 一个任务可 commit 多次（逻辑拆分清晰时）
4. 不附带 Co-Authored-By 签名
5. 更新 `<req-id>.log.md` 记录 commit 信息
6. **暂停确认**（`--auto` 模式除外）

---

## Phase 7: FINISH（里程碑收尾）

当前里程碑所有任务闭环后执行。**这是阶段性标识，不是项目终点。**

1. **质量检查**：lint + typecheck + 调试遗留
2. **性能分析**：针对本次里程碑变更
3. **文档同步**：检查并更新相关文档
4. **Issue 扫描**：汇总 `.claude/issues/` 中待决策的 Issue，输出提醒
5. **里程碑总结**：

```
## 里程碑总结 — [名称]

### 任务完成情况
- ✅ 已完成：X / 共 Y 项
- ⚠️ 延后：X 项（原因）
- 🚫 阻塞：X 项（待人工决策，见 Issue #N）

### 关键改动
1. [改动] — [目的]

### 下一里程碑
[描述 or 「项目完成」]
```

6. **更新 state**：标记里程碑 done，写入下一里程碑信息

**输出后决策**：
- 有后续里程碑 → 回到 PLAN（下一 M）
- 全部完成 → 项目结束
- 发现问题 → 回到 CODE

## Issue 升级机制

当开发过程中遇到**必须人工介入**的问题时：

### AI 自动创建 Issue

文件：`.claude/issues/ISSUE-<序号>-<简短描述>.md`

必须包含：
1. **问题描述**：现象、触发条件、影响范围
2. **当前上下文**：Phase、任务、相关文件、已尝试步骤
3. **方案 A（推荐）**：描述 + 影响 + 风险
4. **方案 B**：描述 + 影响 + 风险
5. **用户方案槽**：空白，供用户填写

### AI 定期检查

每次状态转换（取新任务、进入 REVIEW、进入 FINISH）时：
1. 扫描 `.claude/issues/` 下所有 Issue
2. 检查是否有已决策但未执行的（已选方案但未标记 `已执行`）
3. 发现已决策 Issue → 按选定方案继续推进
4. 汇总待决策 Issue 提醒用户

### 用户主动查看

用户可随时打开 `.claude/issues/` 查看和决策。选择方案 A/B 或填写自己的方案即可。

## 自主决策权限

无需询问，直接执行：

| 场景 | 决策依据 |
|------|---------|
| 代码风格 | ESLint/Prettier |
| 命名 | code-standards |
| 文件组织 | 现有项目结构 |
| 依赖选择 | 优先复用 |
| 类型标注 | 严格 TS，禁止 `any` |
| 错误处理 | 显式，不静默 |
| 文档过期更新 | 自动 w-doc |
| 审查通过 → 测试 | 自动流转 |
| 测试通过 → 提交 | 自动流转 |
| Issue 定期检查 | 每次状态转换 |
| 状态文件写入 | 每次状态转换 |

必须暂停确认（`--auto` 模式除外）：

| 场景 | 原因 |
|------|------|
| PLAN Q&A 对齐后 | 方向需确认 |
| PLAN 四件套产出后 | 方案需确认 |
| DESIGN 风格选择 | 审美需用户决策 |
| 每个 UI 页面生成后 | 逐步确认 |
| 新增第三方依赖 | 需评估 |
| 破坏性 API 变更 | 兼容性策略 |
| 每个任务 COMMIT 前 | 变更确认 |
| 两个方案各有优劣 | 无法自行判断 |
| 里程碑 FINISH 后 | 确认下一里程碑 or 停止 |

```
project/
├── .claude/
│   ├── awf-state.json          # 工作流状态（持久化）
│   ├── user/                   # 用户个性化配置
│   │   ├── README.md
│   │   └── style-preferences.md
│   └── issues/                 # Issue 升级文档
│       ├── ISSUE_TEMPLATE.md
│       └── ISSUE-001-*.md
├── docs/
│   ├── <feature>.md            # 项目级需求文档（PLAN 产出）
│   └── <module>/
│       ├── <req-id>.md         # 需求级需求文档
│       ├── <req-id>.test.md    # 测试用例（721）
│       └── <req-id>.log.md     # 开发日志
└── temp/                       # 临时文件（可随时删除）
    └── review-*.md / test-*.md
```
