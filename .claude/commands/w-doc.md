# 功能模块文档

为指定模块生成或增量更新文档。支持两种粒度：
- **模块级**：完整的模块文档 + 测试用例文档（同之前）
- **需求级**：单个需求的三件套文档（需求文档 + 测试用例 + 开发日志）

自动判断：文档不存在则从零生成，已存在则增量更新。

## 参数

`$ARGUMENTS` — 模块名称或 `模块/需求ID` 格式。

- 模块级：`/w-doc agent` → `docs/agent.md` + `docs/agent.test.md`
- 需求级：`/w-doc agent/req-003` → `docs/agent/req-003.md` + `docs/agent/req-003.test.md` + `docs/agent/req-003.log.md`

可选 flag：`--force` 强制从零重写。

## 需求级文档（三件套）

当 `$ARGUMENTS` 包含 `/` 时，按需求级粒度处理。一个需求 = 一个逻辑集中的任务（对应 WBS 的一个叶子节点）。

### 文件命名约定

```
docs/<module>/
├── <req-id>.md          # 需求文档
├── <req-id>.test.md     # 测试用例（721 渐进式）
└── <req-id>.log.md      # 开发日志
```

### 1. 需求文档（`<req-id>.md`）

```markdown
# [需求名称]（<req-id>）

> 父需求：[链接到规划阶段的需求文档]
> 对应 WBS：[WBS 节点编号]

## 概述
[1-3 句话说明这个需求要做什么]

## 功能描述
[详细的功能说明，包含边界条件和异常情况]

## 涉及文件
| 文件 | 变更类型 | 说明 |
|------|---------|------|
| src/xxx.vue | 修改 | ... |

## 接口/消息协议
[如有新增或修改的 API/消息]

## 验收标准
- [ ] 标准 1
- [ ] 标准 2
```

### 2. 测试用例（`<req-id>.test.md`）

按 **721 渐进式** 组织：

```markdown
# [需求名称] 测试用例

> 测试策略：721 渐进式（70% 单元 + 20% 集成 + 10% E2E）

## 一、单元测试用例（70%）

### TC-1.1 [场景名]
| 项 | 内容 |
|---|---|
| **前置条件** | [精确初始状态] |
| **操作** | [具体调用] |
| **期望** | [精确结果] |

## 二、集成测试用例（20%）

## 三、E2E 测试用例（10%）

## 附录：测试代码触发条件
默认不写测试代码，仅留用例文档。以下情况补充代码：
- 用例无法被正确判定（逻辑过于复杂）
- 频繁触发回归导致 token 消耗过大
```

### 3. 开发日志（`<req-id>.log.md`）

```markdown
# [需求名称] 开发日志

> 记录核心变更和关键决策，不记录临时调试信息。

## 变更记录

| 日期 | 变更类型 | 描述 | 关联 commit |
|------|---------|------|------------|
| ... | feat/fix/refactor | ... | ... |

## 关键决策

1. [决策] — [原因] — [权衡]
2. ...

## 已知问题

- [问题描述] — [影响] — [计划]
```

## 模块级文档

当 `$ARGUMENTS` 不含 `/` 时，按模块级粒度处理（同之前流程）。

### 文件输出

- `docs/<module>.md` — 功能文档
- `docs/<module>.test.md` — 测试用例

详细执行流程见原有的新建/更新流程（步骤一至五）。

## 路由逻辑

1. 解析 `$ARGUMENTS`：
   - 含 `/` → 需求级，解析出 `<module>/<req-id>`
   - 不含 `/` → 模块级
2. 检查目标文件是否存在
3. **不存在** → 走新建流程
4. **已存在 且无 --force** → 走更新流程
5. **已存在 且有 --force** → 走新建流程（覆盖）

## 文档撰写通用原则

1. **代码驱动**：所有描述必须基于实际代码分析
2. **中文优先**：正文用中文，代码标识符保留英文
3. **表格优先**：结构化信息优先使用表格
4. **关注 Why**：记录设计决策和权衡
5. **精确引用**：文件路径 + 函数名
