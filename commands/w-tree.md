# 任务拆解树

将输入描述拆解为**可执行的任务树**，生成双视图（矩形树图总览 + 任务列表）的自包含 HTML 文件在浏览器中展示。

核心目标：面对复杂任务（重构项目、新建项目、大功能开发）时，先拆解再逐个执行。矩形树图用于一屏总览和沟通对齐，任务列表用于查看详情和状态。

## 参数

`$ARGUMENTS` — 要拆解的内容描述。支持多种输入形式：

- 文字需求：`帮我重构云控系统` — 分析代码后拆解为任务树
- 模块/目录路径：`src/background` — 基于代码结构拆解
- 功能描述：`实现灰度引擎` — 拆解为具体实现步骤

示例：
- `/w-tree 重构认证模块` — 分析现有代码，拆解重构任务
- `/w-tree src/sidepanel 性能优化` — 拆解优化任务
- `/w-tree 从零实现一个配置中心` — 拆解新建项目任务

## 层级定义 (L1-L5)

| 级别 | 名称 | 定义 | 示例 |
|------|------|------|------|
| **L1** | 系统层 | 最高级，完整系统，可嵌套子系统 | 云控配置系统 |
| **L2** | 架构层 | 技术架构大块（前端/后端/子系统等） | Background层、状态层 |
| **L3** | 模块层 | 一个独立的技术能力 | 灰度引擎、配置同步 |
| **L4** | 组织层 | 代码的实际组织（按文件/功能分类等） | 拉取函数、Store getters |
| **L5** | 函数层 | 最小可执行单元 | `_rolloutHit()` |

## 节点数据结构

每个任务节点包含以下字段：

```json
{
  "id": "mod-rollout",
  "name": "灰度引擎",
  "level": 3,
  "desc": "实现 _rolloutHit(ratio) 函数，取 mid 末2位十六进制转 percentile，与 ratio 阈值比较；封装三个业务判断函数",
  "deps": ["mod-init"],
  "done": "_rolloutHit 对 ratio=0/100/边界值返回正确结果；三个 isXxxHit 函数可独立调用",
  "ref": ["src/config/remoteConfig.ts", "docs/remote-config.md#灰度计算"],
  "status": "pending",
  "children": []
}
```

### 字段说明

| 字段 | 给谁用 | 必填 | 说明 |
|------|--------|------|------|
| `id` | AI | 是 | 唯一标识，kebab-case，用于 deps 引用 |
| `name` | 人+AI | 是 | 任务名，简短清晰 |
| `level` | 人 | 是 | 层级 1-5，控制显示粒度 |
| `desc` | AI | 是 | **做什么** — 明确的执行指令，不是描述现状 |
| `deps` | AI | 否 | 依赖的任务 id 数组，决定执行顺序 |
| `done` | AI | 是 | **验收条件** — 做到什么程度算完成，AI 自检用 |
| `ref` | AI | 否 | 参考资源 — 相关文件路径、函数名、文档位置 |
| `status` | 人+AI | 是 | `pending` / `active` / `done` |
| `children` | 人+AI | 否 | 子任务数组 |

### desc 写法要求

- **错误**：`基于 mid 末2位 hex 的百分比灰度放量`（描述现状）
- **正确**：`实现 _rolloutHit(ratio) 函数，取 mid 末2位十六进制转 percentile，与 ratio 阈值比较`（执行指令）

### done 写法要求

- 可自动验证的条件优先：`函数返回正确结果`、`类型检查通过`、`测试用例通过`
- 不可自动验证时写人工验收标准：`hover 图片时正确显示/隐藏按钮`

## 执行流程

1. **分析输入**：根据 `$ARGUMENTS` 理解要拆解的目标
   - 如果涉及现有代码 → 先阅读相关源码和文档，理解现状
   - 如果是新建项目 → 根据需求描述规划架构
   - 优先使用 docs/ 中的文档减少源码阅读量

2. **构建任务树**：按 L1→L5 逐级拆解
   - L1-L2 确定系统边界和架构分层
   - L3 拆出独立模块（可并行开发的单元）
   - L4-L5 细化到具体代码组织和函数
   - 每个节点写清 desc（做什么）、done（验收条件）、ref（参考资源）
   - 分析依赖关系填写 deps

3. **生成 HTML**：读取模板 `.claude/templates/w-tree-template.html`，替换两个占位符后写入 `/tmp/w-tree-<timestamp>.html`
   - `{{TITLE}}` → 任务树根节点的 name（如「云控配置系统」）
   - `{{TASK_DATA}}` → 完整的 taskData JSON 对象（即根节点，包含所有 children）
   - **不要修改模板的其他部分**，直接做字符串替换即可

4. **打开浏览器**：`open /tmp/w-tree-<timestamp>.html`

## 注意

- HTML 文件生成到 `/tmp/` 目录，不污染项目
- 如果输入涉及现有代码，优先读 docs/ 文档和 CLAUDE.md 减少上下文消耗
- 拆解粒度以 L3 模块层为核心 — 先对齐模块划分，再细化 L4/L5
- deps 只引用同级或上级节点的 id，不跨级引用子节点
